# Project Workflow Standard

## 默认行为
- 后续所有新项目默认使用本流程。
- 每个项目启动时，用户手动创建 `<project>-old` 与 `<project>-new` 两个目录。
- 所有改动只在 `-new` 进行；确认前不改 `-old`。

## 目录识别规则（强制）
1. AI 开始实施前必须先识别双目录。
2. 若当前目录是 `*-new`，自动匹配同级 `*-old`。
3. 若当前目录是 `*-old`，自动匹配同级 `*-new`。
4. 若目录缺失、命名不符合规则、或匹配不唯一，必须先和用户确认后再继续。

## 强制规则
1. 阶段前信息检查：若信息不足，先输出缺失项、阻塞原因、建议默认方案，等待确认后继续。
2. 技能调用：用户点名技能或任务明显匹配技能时，必须调用并说明用途。
3. 项目启动文档链路：
   - 新项目或结构性新阶段开始前，先要求用户提出产品需求。
   - 文档生成固定顺序：
     - 先 PRD
     - 再设计文档
     - 再技术文档
   - 生成文档时必须分节点与用户对齐，不能一股脑一次性生成完整长文档。
   - 至少要在以下节点确认后，才能继续下一步：
     - 产品定位与目标用户
     - 核心场景与业务闭环
     - 页面与交互方向
     - 技术边界与演进路线
   - 三份文档确认后，必须作为整个项目后续实施、QA、验收和发布前的长期基线。
4. 子代理策略：
   - 小项目：默认单人，先给是否启用建议。
   - 中/大项目：默认主代理负责编排，子代理负责具体执行。
   - 中/大项目启动时，默认先调用 `task-distributor` 做任务拆解，再按边界派单。
   - 子代理优先使用 `C:\Users\29552\.codex\agents` 中的自定义角色，官方内置角色只作兜底。
   - 默认并发强度改为 2 条，不再默认一次性放出 4-6 条。
   - 只有满足以下条件时才允许短时提升到 3 条：
     - 最近没有出现 `429 Too Many Requests`
     - 第 3 个代理是轻量只读分析型
     - 第 3 个代理不阻塞主线
   - 分阶段派单：
     - Phase A：只启动 `task-distributor`
     - Phase B：同时最多 2 个规格锁定代理
     - Phase C：同时最多 2 个写入型代理
     - Phase D：默认由主代理集成，仅在缺口明确时再派 1 个补丁型代理
   - 若首次遇到 `429 Too Many Requests`，立即停止继续 spawn，新建代理前等待 60-90 秒，且只重试一次，并缩小 prompt 上下文。
   - 若第二次遇到 `429 Too Many Requests`，活跃并发降到 1，剩余任务改为顺序执行，并在汇报中明确说明限流角色、降级措施、串行任务和时效影响。
   - `fork_context: true` 不再默认开启，只在确实需要完整线程上下文时使用。
   - 优先传“文件路径 + 事实摘要 + 明确输出物”，而不是整段对话上下文。
   - 优先复用已有子代理线程，而不是重复 spawn 新线程。
   - 已完成或空闲的代理应及时关闭。
   - 若缺少关键子代理角色导致项目无法按既定方式推进，必须先向用户说明缺失角色、阻塞原因、可选降级方案与风险，不能静默降级。
   - 主代理仅在关键路径被阻塞时允许亲自补位，不默认吞掉大块实现工作。
5. UI 改动：先给预览，再改代码。
6. 每轮结束：必须有验证结论（语法/核心路径/关键异常）。
7. 仅在用户确认后执行 `new -> old` 晋升。

## 工作流云端备份（强制）
当以下文件更新后，必须执行 GitHub 备份：
- `WORKFLOW.md`
- `workflow-templates/SKILL.md`
- `workflow-templates/PROMPT.txt`
- `workflow-templates/RETRO_TEMPLATE.md`
- `workflow-templates/WORKFLOW_CHANGELOG.md`
- 当前全局技能 `xiaodu-propmt/SKILL.md`

备份仓库：`git@github.com:xiaodu295521/xiaodu-prompt.git`

备份映射：
- `workflow/WORKFLOW.md`
- `workflow/templates/*`
- `workflow/global-skill/xiaodu-propmt/SKILL.md`
- `workflow/global-agents/AGENTS.md`

备份规则：
1. 若当前项目提供本地 GitHub 上传缓冲区（例如 `<project-root>\github-upload`），先同步工作流快照到缓冲区。
2. `pull -> sync -> diff -> commit -> (用户确认) -> push`
3. 无差异时报告“无备份更新”。
4. push 失败不影响本地流程文件，仅报告并保留本地提交。

## 项目收尾（可演进）
- 项目结束必须复盘：新难点、遗漏点、规则新增/修改/删除建议。
- 按复盘更新流程版本并写入变更记录。

## 每轮固定输出
1. 本轮目标
2. 信息充足性检查
3. 技能使用情况
4. 子代理使用情况
5. 改动摘要
6. 验证结果
7. 风险与待确认项
8. 是否建议晋升（`new -> old`）
