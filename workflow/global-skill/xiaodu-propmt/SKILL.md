# Project Workflow Standard

## 默认行为

- 新项目默认可以采用这套流程
- 用户先创建 `<YOUR_PROJECT>-old` 和 `<YOUR_PROJECT>-new`
- 所有修改只在 `-new` 进行
- `-old` 保存每一轮开始前的完整项目基线

## 目录识别规则（强制）

1. AI 开始实施前先识别双目录
2. 当前目录是 `*-new` 时，自动匹配同级 `*-old`
3. 当前目录是 `*-old` 时，自动匹配同级 `*-new`
4. 如果目录缺失、命名不符合规则，或匹配不唯一，先确认再继续

## `new -> old` 基线快照规则（强制）

每次开始以下工作前，先把 `-new` 的完整项目基线复制到 `-old`：

- 新一轮开发
- 较大修复
- 规则更新
- 大模块改造
- 发布前收口

执行要求：

- 默认覆盖源码、文档、配置和静态资源
- 默认不复制 `node_modules`、构建产物和临时缓存
- 如果这一步没做，不应继续深度修改 `-new`

## 文档链路（强制）

对于新项目或结构性新阶段：

1. 先收集产品需求
2. 生成 PRD
3. 基于 PRD 生成设计文档
4. 基于 PRD 和设计文档生成技术文档
5. 三份文档确认后，再进入实现

继续下一步前，至少对齐：

- 产品定位与目标用户
- 核心场景与业务闭环
- 页面与交互方向
- 技术边界与演进路线

## 用户沟通方式

- 面向用户发送实施计划、阶段汇报、风险说明和技术解释时，默认使用通俗中文
- 优先讲结果、影响和下一步
- 少用术语
- 必须用术语时，补一句白话解释

## 子代理策略

- 小项目：默认单人，按需要决定是否启用子代理
- 中大项目：默认“主代理编排，子代理执行”
- 中大项目启动时，默认先调用 `task-distributor`
- 优先使用你自己环境中的自定义子代理

### 并发规则

- 默认并发 `2`
- 满足以下条件时可短时提升到 `3`：
  - 最近没有 `429 Too Many Requests`
  - 第 3 个代理是轻量只读分析型
  - 第 3 个代理不阻塞主线

### 分阶段派单

- Phase A：只启 `task-distributor`
- Phase B：最多 `2` 个规格锁定代理
- Phase C：最多 `2` 个写入型代理
- Phase D：主代理本地集成，仅在缺口明确时再派 `1` 个补丁代理

### 限流应对

- 首次 `429`：停止继续派单，等待 `60-90` 秒后缩小上下文重试一次
- 第二次 `429`：活跃并发降到 `1`，剩余任务改为顺序执行

### 关键角色缺失

如果缺少关键子代理角色而影响推进，必须先告诉用户：

- 缺的是哪个角色
- 为什么它关键
- 当前还能否继续
- 可选降级方案
- 降级风险

## 规则保存

当这些工作流文件更新后，建议同步到你的规则备份仓库：

- `workflow/WORKFLOW.md`
- `workflow/templates/*`
- `workflow/global-skill/xiaodu-propmt/SKILL.md`
- `workflow/global-agents/AGENTS.md`

远端仓库地址请替换为：

- `<YOUR_BACKUP_REPO_URL>`

默认顺序建议改为：

1. 先同步到本地规则仓库或本地镜像目录
2. 先做本地提交
3. 暂不主动推送到云端仓库
4. 等规则相对稳定后，再由用户确认是否执行远端 push

## 每轮固定输出

1. 本轮目标
2. 是否已完成 `new -> old` 基线快照
3. 信息充足性检查
4. 技能使用情况
5. 子代理使用情况
6. 改动摘要
7. 验证结果
8. 风险与待确认项
