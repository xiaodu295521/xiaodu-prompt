# Project Workflow Standard

## 默认行为

- 这套流程是后续项目的默认基线。
- 项目启动前，先识别项目类型，再套稳定启动方案。
- 项目真正实施前，先确认双目录、文档链路、目录预留和子代理边界。
- 面向用户的说明默认用通俗中文。

## 项目类型分流

### Web 项目

如果项目明显属于以下类型，默认提醒并优先采用 `方案 B：稳定内核 + 可换技术栈`：

- 网站
- Web 应用
- 后台管理
- SaaS
- 落地页
- 仪表盘
- Web 全栈项目

提醒时要明确讲清：

- 为什么建议用它：稳、整齐、后期不容易乱
- 什么是固定的：目录结构、模块边界、contracts、测试门禁、review 规则
- 什么是可变的：技术栈可按项目选

Web 项目默认结构原则：

- 根目录保持克制，不把业务、脚本、配置、文档全部摊平在根目录
- 默认优先使用：
  - `apps/`
  - `packages/`
  - `docs/`
  - `scripts/`
  - `infra/` 或 `config/`
- 前端内部优先使用：
  - `app`
  - `pages` 或 `routes`
  - `features`
  - `entities`
  - `shared`
- 后端内部优先使用：
  - `modules`
  - `adapters`
  - `contracts`
  - `infrastructure`

Web 项目默认 CI/CD 策略：

- 对“一个人 + AI”的 Web 项目，默认尽早启用 CI，不等到项目很大再补。
- 最小 CI 默认包含：
  - 依赖安装通过
  - `lint` 通过
  - `typecheck` 通过
  - `build` 通过
  - 如果已有关键测试，至少跑关键测试
- 默认交付策略为：
  - 开启 CI
  - 开启预览环境
  - 正式环境手动发布
- 不默认把正式环境做成自动发布，除非用户明确要求。
- 对用户解释时，要用通俗中文说明：
  - `CI` 是自动体检
  - 预览环境是先看实际效果的样板间
  - 手动正式发布是最后一步由人拍板，更稳

### 非 Web 项目

非 Web 项目默认走“三分流”：

- 移动应用
  - 默认方案：`app / features / shared / services / docs`
- 桌面工具
  - 默认方案：`shell / core / adapters / docs`
- 自动化 / 机器人 / bridge 服务
  - 默认方案：`workflow / adapters / integrations / runtime / docs`

如果项目跨类型：

- 先按主交付形态选主方案
- 次要部分作为子模块

如果项目类型不清楚：

- 先告诉用户它最像哪两类
- 再让用户拍板

## 双目录规则

- 用户先创建 `<project>-old` 和 `<project>-new`
- 所有实现只在 `-new`
- `-old` 保存“本轮开始前”的完整基线快照

开始以下工作前，必须先做 `new -> old` 基线同步：

- 新一轮开发
- 较大修复
- 规则更新
- 大模块改造
- 发布前收口

默认同步：

- 源码
- 文档
- 配置
- 静态资源

默认不复制：

- `node_modules`
- 构建产物
- 临时缓存

## 文档链路

新项目或结构性新阶段，先走文档链路：

1. PRD
2. 设计文档
3. 技术文档

执行要求：

- 不能一次性把整套长文全部生成完
- 必须分节点和用户对齐
- 至少要先对齐：
  - 产品定位和目标用户
  - 核心场景和主业务链路
  - 页面方向和交互方向
  - 技术边界和演进路线

## 未来扩展对齐

以后写 PRD 和技术文档时，不只对齐当前需求，还要对齐高概率未来需求。

必须额外确认：

- 当前版本一定要做什么
- 下一阶段高概率会加什么
- 哪些能力暂时不做，但目录和边界要先预留
- 哪些外部服务后续可能接入
- 哪些模块未来大概率拆开

PRD 固定新增一段：

- `当前范围 / 下一阶段范围 / 暂不实现但要预留的能力`

技术文档固定新增两段：

- `未来模块演进图`
- `目录预留方案`
- `CI/CD 策略`

如果未来需求还不清楚：

- 至少锁定 1 个“最可能的下一阶段”

## 目录预留规则

目录预留不是可选习惯，而是项目启动流程的一部分。

固定时机：

1. 用户需求初步明确
2. PRD 和技术方向完成一轮对齐
3. 技术文档确认后
4. 再创建项目骨架和预留目录

默认采用 `中等预留`：

- 只预留高概率模块
- 不预留纯猜测模块
- 每个预留目录都必须有明确用途
- 预留数量以“结构清楚”为目标，不以“看起来专业”为目标

## 子代理编排

- 小项目默认单代理
- 中大型项目默认“主代理编排，子代理执行”
- 中大型项目启动时，先调 `task-distributor`
- 优先使用 `C:\Users\29552\.codex\agents` 里的自定义子代理

并发规则：

- 默认并发 `2`
- 只有满足下面条件时，才允许短时升到 `3`
  - 最近没有 `429 Too Many Requests`
  - 第 3 个代理是轻量只读分析型
  - 第 3 个代理不阻塞主线

分阶段派单：

- Phase A：只启 `task-distributor`
- Phase B：最多 `2` 个规格锁定代理
- Phase C：最多 `2` 个写入型代理
- Phase D：主代理本地集成，仅在缺口明确时再派 `1` 个补丁代理

如果缺少关键角色而影响推进，必须先告诉用户：

- 缺的是哪个角色
- 为什么关键
- 能否继续
- 可选降级方案
- 降级风险

## 限流应对

- 首次 `429`
  - 停止继续派单
  - 等待 `60-90` 秒
  - 缩小上下文后只重试 1 次
- 第二次 `429`
  - 并发降到 `1`
  - 剩余任务改为顺序执行
  - 汇报里明确说明影响

## 用户沟通方式

- 默认使用通俗、易懂、结果导向的中文
- 先讲结果
- 再讲原因
- 最后讲实现方式
- 少用术语
- 必须用术语时，顺手补一句白话解释

## 工作流本地保存

工作流文件更新后，默认顺序：

1. 先更新源文件
2. 再同步到本地工作流仓库或本地缓冲区
3. 默认只做本地保存
4. 未经用户明确确认，不主动上传云端仓库

如果当前项目提供本地 GitHub 上传缓冲区，例如 `<project-root>\github-upload`：

- 先同步快照到缓冲区
- 再决定是否做本地提交
- 云端 push 继续等待用户确认

## 每轮固定输出

1. 本轮目标
2. 是否已完成 `new -> old` 基线快照
3. 项目类型与所选稳定方案
4. 信息是否充足
5. 用了哪些技能
6. 派了哪些子代理
7. 这轮改了什么
8. 验证结果
9. 风险和待确认项
