# Global Workflow

这份文件是公开模板库里的全局工作流总览。

它回答三件事：

- 新项目应该怎么启动
- 不同类型项目该套哪种稳定方案
- 文档、目录、子代理和规则快照该怎么维护

## 核心原则

- 先识别项目类型，再进入实施
- 先保留可回退版本，再修改当前工作目录
- 先把需求和未来高概率扩展讲清楚，再写代码
- 主代理负责拆解、协调、集成和汇报
- 子代理负责边界清晰的具体执行
- 面向用户的说明默认用通俗中文
- 规则更新默认先保存在本地，不主动上传云端
- Web 项目默认把 CI/CD 作为工作流的一部分，而不是后期补丁

## 项目类型分流

### Web 项目

如果项目明显属于网站、后台管理、SaaS、落地页、仪表盘或 Web 全栈，默认提醒并优先采用：

- `方案 B：稳定内核 + 可换技术栈`

这套方案的重点是：

- 目录结构固定
- 模块边界固定
- contracts 固定
- 测试和 review 门禁固定
- 技术栈可按项目选

Web 项目默认交付策略：

- 先启用最小 CI
- 再启用预览环境
- 正式环境默认手动发布

适合“一个人 + AI”这种模式的原因是：

- 改动频率高，容易带出回归
- 需要自动体检，但不需要一开始就全自动上线正式环境
- 先看预览，再人工确认发布，更稳

### 非 Web 项目

非 Web 项目默认走三分流：

- 移动应用：`app / features / shared / services / docs`
- 桌面工具：`shell / core / adapters / docs`
- 自动化 / 机器人 / bridge 服务：`workflow / adapters / integrations / runtime / docs`

如果项目跨类型：

- 先按主交付形态选主方案
- 次要部分作为子模块

## 双目录规则

如果项目采用 `<YOUR_PROJECT>-old` / `<YOUR_PROJECT>-new`：

- 所有实际开发只在 `-new`
- `-old` 保存本轮开始前的完整基线
- 每次进入新一轮开发、较大修复、规则更新或发布前收口前，先把 `-new` 的完整内容同步到 `-old`
- 完成同步后，再继续修改 `-new`

## 文档先行规则

当项目处于新启动或结构性新阶段时，先走文档链路：

1. PRD
2. 设计文档
3. 技术文档

执行要求：

- 不能一次性把整套长文档全部生成完
- 必须分节点与用户对齐
- 至少对齐：
  - 产品定位与目标用户
  - 核心场景与主链路
  - 页面与交互方向
  - 技术边界与演进路线

## 未来扩展对齐

PRD 和技术文档阶段，不只写当前版本，还要对齐高概率下一阶段需求。

固定对齐项：

- 当前范围
- 下一阶段范围
- 暂不实现但要预留的能力
- 未来可能接入的外部服务
- 未来可能拆开的模块

固定章节建议：

- PRD：`当前范围 / 下一阶段范围 / 暂不实现但要预留的能力`
- 技术文档：`未来模块演进图`
- 技术文档：`目录预留方案`

## 目录预留规则

目录预留不是可选习惯，而是项目启动的一部分。

默认采用“中等预留”：

- 只预留高概率模块
- 不预留纯猜测模块
- 每个预留目录都必须有明确用途
- 不把大量空目录一次性铺满

## 子代理规则

- 中大型项目默认采用“主代理编排，子代理执行”
- 优先使用自定义子代理
- 默认并发 `2`
- 只有满足条件时短时升到 `3`
- 首次 `429`：停止继续派单，缩小上下文后重试一次
- 第二次 `429`：降为串行

## 规则文件入口

- 全局规则入口：`workflow/global-agents/AGENTS.md`
- 全局技能入口：`workflow/global-skill/xiaodu-propmt/SKILL.md`
- 工作流资产模板：`workflow/templates/*`
- 可复制项目模板：`templates/*`

## 规则保存默认方式

- 工作流规则更新后，先更新源文件
- 再同步到本地规则仓库或本地缓冲区
- 默认只做本地保存，不主动推到云端仓库
- 只有用户明确确认规则已经稳定后，才执行云端 push
