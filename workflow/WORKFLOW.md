# 项目标准工作流（可演进）

版本：`v1.2`  
适用范围：后续所有本地项目（默认启用）

## 1. 核心目标
- 保证每轮改动可回退，避免过度改动导致项目不可用。
- 在信息不足时先确认，不盲目推进。
- 明确技能与子代理使用时机，提升稳定性与效率。
- 每次更新工作流后自动进入 GitHub 云端备份流程。

## 2. 启动前必做
1. 难度评估：`小 / 中 / 大`，说明判断依据。
2. 给出建议：是否启用子代理，建议最小分工。
3. 给出技能计划：本轮预计使用哪些技能及原因。
4. 手动创建双目录（项目名 + 后缀）：
- `<project>-old`：稳定基线（只读保护）
- `<project>-new`：本轮改动区（唯一允许改动位置）
5. AI 启动时必须先识别双目录：
- 若当前在 `*-new`：自动匹配同级 `*-old`
- 若当前在 `*-old`：自动匹配同级 `*-new`
- 若两者缺失或匹配不唯一：先确认，禁止实施

## 3. 双目录安全机制（强制）
1. 每轮开始：从 `-old` 复制到 `-new`。
2. 所有开发、测试、打包在 `-new` 执行。
3. 本轮验证通过前，`-old` 禁止修改。
4. 若本轮失败：丢弃 `-new`，回退使用 `-old`。
5. 若本轮通过且用户确认：`-new -> -old` 晋升，再进入下一轮。

## 4. 信息不足先确认（强制）
每个阶段推进前先做充足性检查；若不足，先输出：
1. 缺失项
2. 为什么阻塞
3. 建议默认方案

仅在用户确认后继续推进。

## 5. 技能调用规则（强制）
1. 用户点名技能：必须调用。
2. 任务明显匹配技能：必须调用。
3. 调用步骤：先读对应 `SKILL.md`，再执行。
4. 每轮汇报中必须说明“使用了哪些技能、用途是什么”。
5. 技能不可用时：明确原因并提供 fallback。
6. 浏览器自动化/站点 QA 优先使用 gstack 相关技能。

## 6. 子代理调用规则
1. 小项目默认单人执行，但先给“是否启用子代理”建议。
2. 中/大项目默认建议启用子代理，并给最小分工。
3. 适合启用：可并行、低耦合、独立模块、评审/验证可拆分。
4. 不适合启用：改动小、同文件高耦合、关键路径强依赖。
5. 使用后必须汇报：代理角色、负责范围、结果摘要。

## 7. UI 工作规则
1. UI 改动前先给预览（Markdown 或本地图片），不直接改代码。
2. 用户确认视觉方向后再开始改动。

## 8. 每轮实施流程
1. 明确本轮目标与验收标准。
2. 在 `-new` 中实施。
3. 执行验证：
- 语法/导入检查
- 核心功能路径
- 关键异常路径
4. 输出验证结论与风险。
5. 请求用户确认是否晋升到 `-old`。

## 9. 打包发布规则
1. 打包前清理旧包。
2. 打包后汇报产物：路径、时间、大小。
3. 若用户要求“仅保留安装包”，删除 `build/spec/cache` 等中间产物。

执行规则：
1. 先 clone/pull 最新 `main`。
2. 同步文件到对应子目录。
3. 若无变更：报告“无备份更新”并结束。
4. 若有变更：先提交本地 commit，展示摘要。
5. push 前必须征得用户确认。
6. 若 push 失败：保留本地 commit，不影响项目本地流程。

## 11. 项目收尾复盘（可演进）
每个项目结束必须执行流程复盘，记录：
1. 新难点
2. 流程遗漏
3. 新增规则建议
4. 删除/简化建议

确认后更新流程版本（如 `v1.2 -> v1.3`），并写入变更记录。

## 12. 每轮固定汇报格式
1. 本轮目标
2. 信息充足性检查
3. 技能使用情况
4. 子代理使用情况
5. 改动摘要
6. 验证结果
7. 风险与待确认项
8. 是否建议晋升（`new -> old`）
