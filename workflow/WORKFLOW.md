# Global Workflow

这份文件是公开模板库里的全局工作流总览。

它回答的是三件事：

- 新项目应该怎么开始
- 多代理任务应该怎么分工
- 工作流规则应该怎么维护和升级

## 核心原则

- 先识别项目目录结构，再进入实施
- 先保留可回退版本，再修改当前工作目录
- 先把需求讲清楚，再写代码
- 主代理负责拆解、协调、集成和汇报
- 子代理负责边界清晰的具体执行
- 面向用户的说明默认尽量说人话
- 规则更新默认先保存在本地仓库，不主动推云端

## 双目录规则

如果项目采用 `<YOUR_PROJECT>-old` / `<YOUR_PROJECT>-new`：

- 所有实际开发只发生在 `-new`
- `-old` 保存本轮开始前的完整基线
- 每次进入新一轮开发、较大修复、规则更新或发布前收口前，先把 `-new` 的完整内容同步到 `-old`
- 完成同步后，再继续修改 `-new`

这一步的目的不是形式化留档，而是给当前轮次保留一个完整可回退版本。

## 文档先行规则

当项目处于新启动或结构性新阶段时，先走文档链路：

1. PRD
2. 设计文档
3. 技术文档

执行要求：

- 不一次性把整套长文档全部生成完
- 必须分节点与用户对齐
- 至少对齐：
  - 产品定位与目标用户
  - 核心场景与主链路
  - 页面与交互方向
  - 技术边界与演进路线

## 用户沟通规则

面向用户发送这些内容时，默认使用通俗、易懂、结果导向的中文：

- 实施计划
- 阶段汇报
- 风险说明
- 技术解释

推荐表达顺序：

1. 会带来什么结果
2. 为什么这样做
3. 技术上怎么实现

## 子代理规则

- 中大项目默认采用“主代理编排，子代理执行”
- 优先使用自定义子代理
- 默认并发为 `2`
- 仅在满足条件时短时提升到 `3`
- 首次 `429`：停止继续派单，缩小上下文后重试一次
- 第二次 `429`：降为串行

## 规则文件入口

- 全局规则入口：`workflow/global-agents/AGENTS.md`
- 全局技能入口：`workflow/global-skill/xiaodu-propmt/SKILL.md`
- 工作流资产模板：`workflow/templates/*`
- 可复制项目模板：`templates/*`

## 规则保存默认方式

- 工作流规则更新后，先更新源文件
- 再同步到本地规则仓库
- 默认只做本地提交，不主动推送到云端仓库
- 只有用户明确确认“这一轮规则已经稳定，可以上传”时，才执行云端 push
