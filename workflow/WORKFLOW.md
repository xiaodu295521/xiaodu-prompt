# Global Workflow

这份文件是当前默认的全局工作流摘要。

## 核心原则

- 先识别项目目录结构，再开始实施
- 先保留可回退版本，再修改当前工作目录
- 先把需求讲清楚，再进入实现
- 主代理负责拆解、协调、集成和汇报
- 子代理负责边界清晰的具体执行
- 面向用户的说明默认使用通俗中文

## 双目录规则

对于采用 `<project>-old` / `<project>-new` 的项目：

- 所有改动只发生在 `-new`
- `-old` 不是空目录占位
- 每次开始新一轮开发、较大修复、规则更新或发布前收口前，先把 `-new` 的完整项目基线复制到 `-old`
- 复制完成后，再继续修改 `-new`

## 文档先行规则

当项目处于新启动或结构性新阶段时，先走文档链路：

1. PRD
2. 设计文档
3. 技术文档

执行要求：

- 不一次性把整套长文档全部生成完
- 必须分节点与用户对齐
- 至少对齐：
  - 产品定位与目标用户
  - 核心场景与主链路
  - 页面与交互方向
  - 技术边界与演进路线

## 用户沟通规则

面向用户的这些内容，默认用通俗易懂的中文：

- 实施计划
- 阶段汇报
- 风险说明
- 技术解释

表达顺序优先：

1. 做这件事会带来什么结果
2. 为什么这样做
3. 技术上如何实现

## 子代理规则

- 中大项目默认“主代理编排，子代理执行”
- 优先使用自定义子代理
- 默认并发为 `2`
- 仅在满足条件时短时提升到 `3`
- 首次 `429`：等待并缩小上下文后重试一次
- 第二次 `429`：降为串行

## GitHub 备份规则

- 如果项目有本地上传缓冲区，例如 `github-upload`，先同步快照到缓冲区
- 再进行本地 git 提交和云端 push
- 当前云端备份仓库：`git@github.com:xiaodu295521/xiaodu-prompt.git`
