## Skillify Project Rules

Skillify 项目默认遵守全局 `xiaodu-propmt` 工作流，并在此基础上增加本项目自己的执行规则。

### 项目类型与默认启动方案

- Skillify 当前属于 Web 项目。
- 因此本项目继承全局默认 `方案 B：稳定内核 + 可换技术栈`。
- 对 Skillify 来说，这意味着：
  - 目录结构要保持克制，不把新功能继续堆到根目录
  - 新业务模块优先按领域拆到 `src` 内部，而不是横向散落
  - 后续若接真实服务，优先先锁 contracts、模块边界和目录归属，再动实现

### 目录规则

- 项目使用 `Skillify-old` 和 `Skillify-new` 双目录。
- 所有实际开发只发生在 `Skillify-new`。
- `Skillify-old` 不是空目录占位，而是“本轮开始前”的完整项目快照。
- 每次进入新的开发轮次、需求实现、规则更新或较大改动前，必须先把 `Skillify-new` 的完整项目同步到 `Skillify-old`，再继续修改 `Skillify-new`。
- 如果 `old` 同步还没完成，不应直接进入 `new` 的后续修改。
- 如果目录缺失、命名异常或目标不明确，必须先说明再继续。

### 基线快照规则

- `Skillify-old` 保存的是“本轮开始前”的完整项目版本。
- 默认应覆盖源码、文档、配置和静态资源。
- 默认不需要复制 `node_modules`、构建产物和临时缓存目录。
- 每轮汇报里应明确说明，这轮是否已经完成 `new -> old` 基线同步。

### 文档基线规则

- 正式基线文档位于 `docs` 目录。
- 三份核心文档如下：
  - `docs/01-product-requirements.md`
  - `docs/02-ui-design-spec.md`
  - `docs/03-technical-architecture.md`
- 产品文档定义做什么、不做什么。
- UI 文档定义页面如何呈现和交互。
- 技术文档定义如何实现，以及当前和未来的技术边界。

### 项目启动文档规则

- 新项目、新模块或大阶段启动前，必须先收集产品需求，不能直接开做。
- 文档生成顺序固定为：
  1. 先形成产品需求文档（PRD）。
  2. 再基于 PRD 形成设计文档。
  3. 再基于 PRD 和设计文档形成技术文档。
- 不能一次性把整套长文全部抛给用户。
- 必须按关键节点逐步对齐，例如：
  - 产品定位和目标用户
  - 核心场景和主链路
  - 页面方向和交互方向
  - 技术边界和演进路线
- 只有关键节点对齐后，才能继续下一份文档或进入实施。
- 三份文档确认后，进入 `docs` 基线，作为后续开发、QA 和迭代的事实依据。

### 未来扩展对齐规则

- 写 PRD 和技术文档时，不能只讲当前需求。
- 还要和用户对齐后续高概率会出现的开发需求，包括：
  - 当前版本一定要做什么
  - 下一阶段最可能加什么
  - 哪些能力暂时不做，但目录和边界要先预留
  - 哪些外部服务未来可能接入
  - 哪些模块以后大概率拆开
- PRD 必须新增一段：
  - `当前范围 / 下一阶段范围 / 暂不实现但要预留的能力`
- 技术文档必须新增两段：
  - `未来模块演进图`
  - `目录预留方案`

### 目录预留规则

- 目录预留是项目启动的一部分，不是可选习惯。
- Skillify 以及后续网站类项目默认采用“中等预留”：
  - 只预留高概率模块
  - 不预留纯猜测模块
  - 每个预留目录都必须有明确用途
  - 不把几十个空目录一次性铺满
- 对当前 Web 项目，后续新增高概率模块时，优先考虑在 `src` 下按职责预留：
  - `src/features/<domain>`
  - `src/entities/<domain>`
  - `src/services`
  - `src/shared`
- 不允许把后续功能继续无边界地堆在根目录或单一目录。

### Web 项目 CI/CD 规则

- 对“一个人 + AI”推进的网站项目，默认应尽早启用 CI，不等到项目很大再补。
- 这里的 CI 可以理解成“每次改完代码都先自动体检一次”。
- Web 项目默认最小 CI 门禁包括：
  - 依赖安装通过
  - `lint` 通过
  - `typecheck` 通过
  - `build` 通过
  - 如果仓库里已经有关键测试，则至少跑关键测试
- Web 项目默认发布策略不是“全自动上线正式环境”，而是：
  - 开启 CI
  - 开启预览环境
  - 正式环境保持手动确认发布
- 预览环境的作用是让用户和主代理先看真实页面效果，再决定是否进入正式发布。
- 如果没有用户明确确认，不默认把正式环境改成自动发布。
- 进入发布、验收或接入真实外部服务前，主代理应补充说明：
  - 当前有没有 CI
  - 当前有没有预览环境
  - 正式发布是手动还是自动
  - 还缺哪些交付门禁

### 用户沟通规则

- 面向用户发送实施计划、开发说明、阶段汇报和风险说明时，默认使用通俗易懂的中文。
- 不要把说明写成堆满术语的工程文。
- 如果必须提到技术名词，要立刻补一句“它是做什么的”或“它对用户意味着什么”。
- 优先解释结果、影响、风险和下一步，而不是内部实现细节。
- 当用户明确表示自己不熟悉技术时，要进一步降低表达门槛。

### 重要里程碑回看规则

- 重要里程碑开始前，主代理必须重新读取三份基线文档。
- 重要里程碑固定为：
  - 新的大功能或新页面簇启动前
  - 涉及架构边界变化的实现前
  - 接入真实外部服务前
  - QA、验收、准备发布前
- 每次里程碑汇报必须说明：
  - 本轮引用了哪些基线文档
  - 是否发现产品、设计、技术之间不一致
  - 如果不一致，应更新哪份文档

### 编排规则

- 中大型任务默认采用“主代理编排，子代理执行”。
- 主代理负责：
  - 需求落地
  - 任务拆解 kickoff
  - 依赖控制
  - 结果集成
  - 最终验证
  - 对用户汇报
- 主代理不应默认吞掉大块实现，只在关键路径被阻塞时补位。

### 子代理优先级

- 优先使用 `C:\Users\29552\.codex\agents` 里的自定义角色。
- 官方内置角色只在没有合适自定义角色时兜底。
- 不能静默用低匹配角色替代关键角色。

### 关键角色规则

- 如果当前任务明显依赖关键角色，而该角色缺失或不可用到影响推进，必须先向用户说明：
  - 缺的是哪个角色
  - 为什么它是当前任务所必需的
  - 当前能否继续推进
  - 可以采用什么降级方案
  - 这样做有什么风险

### 并发规则

- 中大型任务默认并发 2 条子任务。
- 只有满足下面三个条件时，才允许短时提升到 3：
  - 最近没有出现 `429 Too Many Requests`
  - 第 3 个代理是轻量只读分析型
  - 第 3 个代理不阻塞主线
- 只允许并行写入范围不重叠的任务。

### 分阶段派单规则

- Phase A：只启 `task-distributor`
- Phase B：同时最多 2 个规格锁定代理
- Phase C：同时最多 2 个写入型代理
- Phase D：默认本地集成，只在缺口明确时再派 1 个补丁代理

### 限流应对规则

- 首次遇到 `429 Too Many Requests`
  - 立即停止继续 spawn 新代理
  - 等待 `60-90` 秒
  - 只重试 1 次
  - 重试时缩小 prompt 上下文
- 第二次遇到 `429 Too Many Requests`
  - 活跃并发立即降到 `1`
  - 余下任务改为顺序执行
  - 汇报里明确说明限流角色、降级措施和时效影响

### Prompt 与线程复用规则

- `fork_context: true` 不再默认开启。
- 优先传“文件路径 + 事实摘要 + 明确输出物”。
- 已存在的子代理优先复用，不要重复 spawn。
- 完成或空闲的代理应及时关闭。

### 工作流快照规则

- `D:\Skillify\github-upload` 是本地 GitHub 上传缓冲区。
- 工作流或规则文件更新后，顺序固定为：
  1. 先更新源文件
  2. 再同步到 `github-upload`
  3. 默认只做本地保存
  4. 只有用户明确确认后，才同步到云端仓库

### 开发进度存档规则

- 项目主进度存档文件固定为 `DEVELOPMENT_PROGRESS_ARCHIVE.md`。
- 后续桌面端会话进入项目后，凡是涉及开发、规则、验证、QA、阶段汇报或风险判断，默认先读取这个文件的最后一条记录，再继续工作。
- 每次对话结束前，如果本轮出现了新的仓库事实，必须把本轮进度追加写入 `DEVELOPMENT_PROGRESS_ARCHIVE.md`。
- 记录内容必须基于真实文件、真实命令结果、真实验证结果，不得用聊天记忆代替仓库事实。
- 若本轮没有改代码，也要记录检查了什么、确认了什么、阻塞在哪里。

### 双端交接规则

- 当前状态主入口固定为 `CURRENT_HANDOFF.md`。
- 历史事实存档固定为 `DEVELOPMENT_PROGRESS_ARCHIVE.md`。
- 桌面端进入项目时，固定顺序为：
  1. 先看 `CURRENT_HANDOFF.md`
  2. 再看 `DEVELOPMENT_PROGRESS_ARCHIVE.md` 最后一条
  3. 再看相关代码、文档、日志
- 移动端进入项目时，固定顺序为：
  1. 先看 `CURRENT_HANDOFF.md`
  2. 只在需要上下文时，再看 `DEVELOPMENT_PROGRESS_ARCHIVE.md` 最近 1-3 条
- 任一端完成实质工作后，更新顺序固定为：
  1. 先更新 `CURRENT_HANDOFF.md`
  2. 再把本轮事实追加到 `DEVELOPMENT_PROGRESS_ARCHIVE.md`
- 如果两份文档不一致，以 `CURRENT_HANDOFF.md` 的当前状态为准，以 `DEVELOPMENT_PROGRESS_ARCHIVE.md` 的历史记录为证据。
- 如果只是临时判断、提问或阻塞，没有形成新的历史事实，也应先更新 `CURRENT_HANDOFF.md` 的“正在处理 / 风险与阻塞 / 下一步”。
