## Skillify Project Rules

Skillify 项目默认遵守 `xiaodu-propmt` 工作流，并在此基础上增加本项目自己的执行规则。

### 目录规则

- 项目使用 `Skillify-old` 和 `Skillify-new` 双目录。
- 所有实际开发只发生在 `Skillify-new`。
- `Skillify-old` 不再作为空目录占位。
- 从现在开始，每次进入新的开发轮次、需求实现、规则更新或较大改动前，必须先把 `Skillify-new` 的完整项目基线覆盖同步到 `Skillify-old`，再继续修改 `Skillify-new`。
- 这一步的目标是保留“开工前的完整可回退版本”，而不是只保留一个空壳目录。
- 如果 `old` 同步没有完成，不应直接进入 `new` 的后续修改。
- 若目录缺失、命名异常或目标不明确，必须先说明再继续。

### `old` 基线快照规则

- `Skillify-old` 保存的是“本轮开始前”的完整项目快照。
- 默认应覆盖源码、文档、配置和静态资源。
- 默认不需要把 `node_modules`、构建产物、临时缓存目录一并复制到 `old`。
- 如果用户明确要求保留某些额外目录，再单独说明。
- 每轮汇报里应说明这轮是否已经完成 `new -> old` 基线同步。

### 文档基线规则

- 正式基线文档位于 `docs` 目录。
- 三份核心基线文档如下：
  - `docs/01-product-requirements.md`
  - `docs/02-ui-design-spec.md`
  - `docs/03-technical-architecture.md`
- 产品文档定义做什么、不做什么。
- UI 文档定义页面如何呈现和交互。
- 技术文档定义如何实现，以及当前和未来的技术边界。

### 项目启动文档规则

- 新项目、新模块或大阶段启动前，必须先收集产品需求，不能直接开做。
- 文档生成顺序固定为：
  1. 先形成产品需求文档（PRD）
  2. 再基于 PRD 形成设计文档
  3. 再基于 PRD 和设计文档形成技术文档
- 生成文档时不能一次性把整套长文全部倒给用户。
- 必须按关键节点和用户逐步对齐，例如：
  - 产品定位和目标用户
  - 核心场景和主链路
  - 页面方向和交互方向
  - 技术边界和演进路线
- 只有关键节点对齐后，才能继续下一份文档或进入实施。
- 三份文档确认后，进入 `docs` 基线，作为后续开发、QA 和迭代的事实依据。

### 用户沟通规则

- 面向用户发送实施计划、开发说明、阶段汇报和风险说明时，默认使用通俗易懂的中文。
- 不要把说明写成堆术语的工程文。
- 如果必须提到技术名词，要立刻补一句“它的作用是什么”或“它对用户意味着什么”。
- 优先解释结果、影响、风险和下一步，而不是内部实现细节。
- 当用户明确表示自己不熟悉技术时，应进一步降低表达门槛。

### 重要里程碑回看规则

- 重要里程碑开始前，主代理必须重新读取三份基线文档。
- 重要里程碑固定为：
  - 新的大功能或新页面簇启动前
  - 涉及架构边界变化的实现前
  - 接入真实外部服务前
  - QA、验收、准备发布前
- 每次里程碑汇报必须说明：
  - 本轮引用了哪些基线文档
  - 是否发现产品、设计、技术之间不一致
  - 若不一致，应更新哪份文档

### 编排规则

- 中大型任务默认采用“主代理编排，子代理执行”。
- 主代理负责：
  - 需求落地
  - 任务拆解 kickoff
  - 依赖控制
  - 结果集成
  - 最终验证
  - 用户汇报
- 主代理不应默认吞掉大块实现，只在关键路径阻塞时补位。

### 子代理优先级

- 优先使用 `C:\Users\29552\.codex\agents` 中的自定义角色。
- 官方内置角色只在没有合适自定义角色时兜底。
- 不得静默用弱匹配角色替代关键角色。

### 关键角色规则

- 如果当前任务明显依赖关键角色，而该角色缺失或不可用到影响推进，必须先向用户说明：
  - 缺的是哪个角色
  - 为什么它是关键角色
  - 当前能否继续推进
  - 可选降级方案是什么
  - 这样做的风险是什么

### 并发规则

- 中大型任务默认并发 `2` 条子任务。
- 只有满足条件时，才允许短时提升到 `3`：
  - 最近没有出现 `429 Too Many Requests`
  - 第 3 个代理是轻量只读分析型
  - 第 3 个代理不阻塞主线
- 只并行写入范围不重叠的任务。

### 分阶段派单规则

- Phase A：只启 `task-distributor`
- Phase B：同时最多 `2` 个规格锁定代理
- Phase C：同时最多 `2` 个写入型代理
- Phase D：默认本地集成，只有缺口明确时再派 `1` 个补丁代理

### 限流应对规则

- 首次遇到 `429 Too Many Requests`
  - 立即停止继续 spawn 新代理
  - 等待 `60-90` 秒
  - 只重试 `1` 次
  - 重试时缩小 prompt 上下文
- 第二次遇到 `429 Too Many Requests`
  - 活跃并发立即降到 `1`
  - 余下任务改为顺序执行
  - 汇报中明确说明限流角色、降级措施、串行影响

### Prompt 与线程复用规则

- `fork_context: true` 不再默认开启。
- 优先传“文件路径 + 事实摘要 + 明确输出物”。
- 已存在的子代理优先复用，不要重复 spawn。
- 完成或空闲的代理应及时关闭。

### GitHub 缓冲区规则

- `D:\Skillify\github-upload` 是 GitHub 上传缓冲区。
- 工作流规则更新后，先更新源文件，再同步到 `github-upload` 快照。
- 缓冲区只保存待上传快照，不作为主开发目录。

### 技能使用

- 优先使用 gstack 相关技能做结构化规划、评审和 QA。
- 常用顺序：
  - `gstack-office-hours`
  - `gstack-plan-ceo-review`
  - `gstack-plan-eng-review`
  - `gstack-plan-design-review`
  - `gstack-review`
  - `gstack-qa` 或 `gstack-qa-only`

### 每轮输出

- 每一轮较大的工作都应汇报：
  1. 本轮目标
  2. 是否已完成 `new -> old` 基线同步
  3. 信息是否充足
  4. 用了哪些技能
  5. 派了哪些子代理
  6. 改了什么
  7. 验证结果
  8. 风险和待确认项

### 工作流文件变更

- 如果 `WORKFLOW.md`、`AGENTS.md` 或其他流程说明被更新：
  - 先更新源文件
  - 再同步到 `D:\Skillify\github-upload`
  - 之后再按既有规则决定是否同步到云端仓库
