## Skillify Project Rules

Skillify 在 `xiaodu-propmt` 工作流下开发。

### 目录规则

- 项目使用 `Skillify-old` 与 `Skillify-new` 双目录。
- 只允许在 `Skillify-new` 中实施改动。
- 未经用户明确确认，不得编辑 `Skillify-old`。
- 若目录对缺失、命名异常或目标不明确，必须先报告。

### 文档基线规则

- 项目正式基线文档位于 `docs` 目录。
- 三份核心基线文档如下：
  - `docs/01-product-requirements.md`
  - `docs/02-ui-design-spec.md`
  - `docs/03-technical-architecture.md`
- 产品文档定义做什么、不做什么。
- UI 文档定义页面如何表达和交互。
- 技术文档定义如何实现以及实现边界。

### 项目启动文档生成规则

- 新项目或新阶段启动前，主代理必须先要求用户提出产品需求，而不是直接进入实现。
- 文档生成顺序固定为：
  1. 先形成产品需求文档（PRD）
  2. 再基于 PRD 形成设计文档
  3. 再基于 PRD 与设计文档形成技术文档
- 生成文档时不能一次性把整套长文档直接倾倒给用户。
- 必须按关键节点与用户对齐，例如：
  - 产品定位与目标用户
  - 核心场景与主链路
  - 页面与交互方向
  - 技术边界与演进路线
- 只有在关键节点对齐后，才能继续生成下一份文档或进入实施。
- 三份文档一旦确认，就进入 `docs` 基线，作为后续项目实施、验收和演进的事实源。

### 重要里程碑回看规则

- 重要里程碑开始前，主代理必须重新读取三份基线文档。
- 重要里程碑固定为：
  - 新的大功能或新页面簇启动前
  - 涉及架构边界变化的实现前
  - 接入真实外部服务前
  - QA、验收、准备发布前
- 每次里程碑汇报必须说明：
  - 本轮引用了哪些基线文档
  - 是否发现产品、设计、技术之间不一致
  - 若不一致，应更新哪份文档
- 若产品需求发生结构性变化，必须先更新 PRD，再回写设计文档和技术文档。

### 编排规则

- 中大型任务默认采用“主代理编排，子代理执行”。
- 主代理负责：
  - 需求 grounding
  - 任务拆解 kickoff
  - 依赖控制
  - 结果集成
  - 最终验证
  - 用户汇报
- 主代理不应默认吞掉大块实现，只在关键路径阻塞时补位。

### 子代理优先级

- 优先使用 `C:\Users\29552\.codex\agents` 中的自定义角色。
- 官方内置角色只在没有合适自定义角色时兜底。
- 不得静默用弱匹配角色替代关键角色。

### 关键角色规则

- 如果当前任务明显依赖关键角色，而该角色缺失或不可用到影响推进，必须先向用户说明：
  - 缺的是哪个角色
  - 为什么它是关键角色
  - 当前能否继续推进
  - 可选降级方案是什么
  - 这样做的风险是什么

### 并发规则

- 中大型任务默认并发 `2` 条子任务，不再默认一次性放出 `4-6` 条。
- 只有满足条件时才允许短时提升到 `3`：
  - 最近没有出现 `429 Too Many Requests`
  - 第 3 个代理是轻量只读分析型
  - 第 3 个代理不阻塞主线
- 只并行写入范围不重叠的任务。
- UI 设计与前端实现不得同时写同一路径。
- 前后端可并行，但接口边界要先锁定。

### 分阶段派单规则

- Phase A：拆解阶段
  - 只启动 `task-distributor`
  - 不与其他规划代理并发
- Phase B：规格锁定阶段
  - 同时最多 `2` 个子代理
  - 推荐组合：
    - `product-manager` + `ui-designer`
    - `product-manager` + `api-designer`
    - `business-analyst` + `code-mapper`
- Phase C：实现阶段
  - 同时最多 `2` 个写入型子代理
  - 推荐组合：
    - 一个前端实现者
    - 一个后端或支付实现者
  - 同一模块、同一路径、同一页面簇禁止并发写入
- Phase D：集成阶段
  - 默认不再继续并发派单
  - 主代理负责收口、验证和冲突处理
  - 只在发现明确缺口时再派 `1` 个补丁型代理

### 限流应对规则

- 首次遇到 `429 Too Many Requests`
  - 立即停止继续 spawn 新代理
  - 等待 `60-90` 秒
  - 只重试 `1` 次
  - 重试时必须缩小 prompt 上下文，避免继续大段 `fork_context`
- 第二次遇到 `429 Too Many Requests`
  - 活跃并发立即降到 `1`
  - 余下任务改为顺序执行
  - 主代理汇报中必须明确说明：
    - 哪个角色限流
    - 已采取的降级措施
    - 哪些任务改为串行
    - 对时效的影响
- 不允许：
  - 继续批量 spawn
  - 静默删除子任务
  - 让主代理无边界接管全部大任务实现

### Prompt 与会话复用规则

- `fork_context: true` 不再默认开启，只在确实需要完整线程上下文时使用。
- 优先传“文件路径 + 事实摘要 + 明确输出物”，而不是整段对话上下文。
- 对已存在的子代理，优先 `send_input` 复用，而不是重复 spawn 新线程。
- 每个子代理只承担一个边界清晰的任务。
- 已完成或空闲的代理要及时关闭，避免长期占用线程槽位。

### GitHub 缓冲区规则

- 项目根目录中的 `D:\Skillify\github-upload` 是 GitHub 上传缓冲区。
- 工作流规则更新后，先更新源文件，再同步到该缓冲区对应快照。
- 缓冲区只保存待上传快照，不作为主开发目录。

### 技能使用

- 优先使用 gstack 相关技能进行结构化规划、设计评审、工程评审、QA 和浏览器验证。
- 常用顺序：
  - `gstack-office-hours`
  - `gstack-plan-ceo-review`
  - `gstack-plan-eng-review`
  - `gstack-plan-design-review`
  - `gstack-review`
  - `gstack-qa` 或 `gstack-qa-only`

### 每轮输出

- 每一轮较大的工作都应汇报：
  1. 本轮目标
  2. 信息充足性检查
  3. 技能使用情况
  4. 子代理使用情况
  5. 改动摘要
  6. 验证结果
  7. 风险与待确认项
  8. 是否建议晋升 `new -> old`

### 工作流文件变更

- 若本项目的 `WORKFLOW.md` 或其他工作流说明被更新，后续需按 `xiaodu-propmt` 规则同步到 `xiaodu-prompt` 备份仓库。
- 同步到外部备份仓库前，先同步到 `D:\Skillify\github-upload` 中的对应快照。
- 备份时先提交，本地同步后再等待用户确认是否 push。
