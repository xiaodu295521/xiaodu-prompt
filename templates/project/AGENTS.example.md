## <YOUR_PROJECT> Project Rules

### 项目类型与启动方案

- 先判断当前项目是不是 Web 项目。
- 如果是 Web 项目，默认继承 `方案 B：稳定内核 + 可换技术栈`。
- 如果不是 Web 项目，按移动、桌面、自动化 / bridge 三分流选择稳定方案。
- 如果是 Web 项目，默认交付策略为 `最小 CI + 预览环境 + 手动正式发布`。

### 目录规则

- 项目使用 `<YOUR_PROJECT>-old` 和 `<YOUR_PROJECT>-new`
- 所有开发只在 `-new`
- 每轮开发开始前，先把 `-new` 的完整项目基线同步到 `-old`

### 文档规则

- 新阶段先做 PRD
- 再做设计文档
- 再做技术文档
- 三份文档确认后，再进入实施

### 未来扩展对齐

- PRD 阶段必须写：
  - 当前范围
  - 下一阶段范围
  - 暂不实现但要预留的能力
- 技术文档阶段必须写：
  - 未来模块演进图
  - 目录预留方案
  - CI/CD 策略

### 目录预留规则

- 默认采用中等预留
- 只预留高概率模块
- 不预留纯猜测模块
- 每个预留目录都必须有用途说明
- 不允许后续功能继续全部堆在根目录或单一目录

### 用户沟通规则

- 计划、汇报、风险说明默认使用通俗中文
- 必须用术语时补一句白话解释

### 子代理规则

- 中大型任务优先由主代理编排
- 默认并发 `2`
- 写入范围重叠时不并行
